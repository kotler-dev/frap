Проблема

Интерфейс меняется быстрее, чем обновляются тесты

Что болит у команд

  • Часы ручного разбора DOM на новых страницах.
  • Рефакторинг фронта вызывает массовые падения селекторов в CI.
  • Непонятно, почему автолечение выбрало новый элемент.
  • Непрозрачные облачные инструменты часто не проходят требования безопасности enterprise.

Чего ждут автоматизаторы

  • Снять снимок страницы и получить все доступные локаторы.
  • Группировать элементы по структуре и бизнес-зонам.
  • Быстро стартовать PageObject без ручной рутины.
  • Снижать нестабильные падения (flaky) за счет детерминированной логики.

Основной цикл

Один жизненный цикл локатора

1. Снять снимок страницы
Собираем карту элементов: теги, атрибуты, путь, роль, текст.
2. Построить сигнатуру
Фиксируем устойчивые признаки и структурный префикс.
3. Поиск и разрешение
Ищем лучшего кандидата по confidence и правилам выбора.
4. Отчет
Формируем объяснимые отличия: healed, confidence, топ кандидатов.
Поиск структуры

MVP для старта: получить все локаторы страницы за один проход

Сценарий "новый продукт, ноль тестов": discovery -> фильтрация -> экспорт -> baseline.

// Java пример: поиск структуры и экспорт MapDiscovery map = frap.discover(page, DiscoverOptions.builder() .scope("interactive") .build()); LocatorExport export = map.export(ExportOptions.builder() .formats(List.of("css", "xpath", "frap-id")) .withConfidence(true) .withClusters(true) .build()); // результат: карта элементов, кластеры и локаторы-кандидаты
Кластеризация

Что такое кластер и зачем он нужен

ClusterType (v1.0.0)

  • Кластер — группа элементов с одной структурной сигнатурой (один шаблон: карточки, строки списка).
  • SINGLE — один элемент (уникальная кнопка, поле).
  • LIST — ≥ 2 похожих (повторяющийся блок).
  • 5 одинаковых карточек → один метод items(int index), а не 5 копий.

Почему не querySelector('*')

  • Тысячи узлов: шум (div, обёртки), не только интерактив.
  • Нет семантики «список vs уникальная кнопка».
  • Нет рекомендованных локаторов и confidence.
  • Discover сужает выборку (интерактив + data-testid в SnapshotBuilder).
  • Кластеризация группирует похожее для PO, heal и filter (min_cluster_size, LIST).

Подробнее: docs/clustering.md · алгоритм Drain3: project/architecture/clustering.md

Кластеризация

Пример: id мигрирует в data-id внутри одного списка

До изменения DOM

  • ul > li#1
  • ul > li#2
  • Оба элемента в одном контейнере и одном sibling-наборе.
  • Матчинг обычно опирается на структуру + атрибуты.

После изменения DOM

  • ul > li#1 остается без изменений.
  • ul > li[data-id='2'] заменяет старый li#2.
  • Кластер должен остаться тем же: это тот же список.
  • Ожидаемый результат: лучший кандидат — li[data-id='2'].

Контракт в fixtures/contract/clustering-id-migration проверяется в Rust/TS/Java: кластер списка сохраняется, heal выбирает li[data-id='2'] при потере li#2.

Healing

Heal: как библиотека восстанавливает локатор

1. Локатор не найден
[data-testid='talk-open-healing'] не найден после изменения разметки.
2. Сопоставление сигнатуры с текущей страницей
Frap берет сохраненную сигнатуру элемента и сравнивает с кандидатами в новом snapshot.
3. Выбор лучшего кандидата по confidence
Лучший кандидат: [data-testid='talk-card-open-healing'] (например, confidence 0.94).
4. Результат
Если confidence выше порога — healed=true, иначе безопасный fail с отчетом.

Heal — это управляемое восстановление локатора, а не угадывание.

UX автоматизатора

Фрагменты кода из Conference demo (Java + отчет)

// ScheduleHealingTest.java FrapLocator openLink = Frap.withFrap( page.locator("[data-testid='talk-open-healing']"), page, ConferencePaths.confFrap().minConfidence(0.7).healPolicy("expect_heal").debug(true) ); openLink.click(); // проверка результата healing HealResult result = Frap.getLastHealResult(openLink); assertThat(result.healed()).isTrue(); assertThat(result.confidence()).isGreaterThanOrEqualTo(0.7);

Показываем в демо связку: `schedule-heal.html` -> `ScheduleHealingTest` -> `frap-debug.html` (DOM clusters, лучшие кандидаты, решение).

PageObject

Быстрый старт первых тестов

1. Поиск структуры

Снять снимок страницы и получить интерактивную карту элементов.

2. Группировка

Выбрать кластеры и важные бизнес-зоны страницы.

3. Экспорт

Сгенерировать локаторы в css/xpath/frap-id и baseline.

4. Генерация

Создать скелет PageObject и типовые методы действий.

5. Проверка

Проверить confidence и пометить рискованные элементы.

6. Развитие

Отслеживать изменения интерфейса и поддерживать тесты без ручного перебора DOM.

Крупные проекты

Что критично для больших проектов (тысячи тестов)

Технические требования

  • Snapshot caching и инкрементальный refresh.
  • Policy-режимы: strict fail / controlled heal.
  • Детерминированное поведение без облачных зависимостей.
  • Интеграция с CI артефактами и отчетами.

Метрики ценности

  • Heal rate и доля ambiguous failures.
  • Динамика изменений интерфейса по страницам и командам.
  • Среднее время восстановления селектора.
  • Доля "ручных DOM правок", которую удалось убрать.
Выбор архитектуры

Rust-ядро + SDK под стек: зачем так, а не копии алгоритмов

Плюсы единого ядра

  • Единый источник истины для signature/heal/clustering.
  • Одинаковое поведение в Java, TypeScript и других SDK.
  • Производительность и детерминизм без расхождений.

Как снизить vendor lock-in

  • Явный JSON-RPC контракт и DTO-схема.
  • Тесты проверки соответствия для каждого SDK.
  • Тонкие адаптеры: API-удобство без копирования логики.
План развития

Финальное видение

Frap = детерминированный движок жизненного цикла локаторов: обнаружение структуры -> стабильные идентификаторы -> управляемый heal -> аналитика изменений интерфейса.

v1
Поиск структуры + сигнатуры + heal + отчеты
v2
Изменения страницы + поиск сначала в кластере
v3
Генерация PO + расширенные API для фреймворков

Для тестировщиков и команд разработки фреймворка: быстрее старт, меньше flaky, больше объяснимости.
