What teams struggle with

  • Hours of manual DOM inspection on new pages.
  • Frontend refactors cause mass selector failures in CI.
  • Unclear why auto-healing picked a new element.
  • Opaque cloud tools often fail enterprise security requirements.

What automators expect

  • Snapshot a page and get all available locators.
  • Group elements by structure and business zones.
  • Start Page Objects quickly without manual routine.
  • Reduce flaky failures with deterministic logic.

Clustering

What a cluster is and why it matters

ClusterType (v1.0.0)

  • Cluster — elements sharing one structural signature (cards, list rows).
  • SINGLE — one element (unique button, field).
  • LIST — ≥ 2 similar items (repeating block).
  • 5 identical cards → one items(int index) method, not 5 copies.

Why not querySelector('*')

  • Thousands of nodes: noise (divs, wrappers), not just interactive.
  • No semantics for “list vs unique button”.
  • No recommended locators or confidence scores.
  • Discover narrows the set (interactive + data-testid in SnapshotBuilder).
  • Clustering groups similar items for PO, heal, and filter (min_cluster_size, LIST).

Details: docs/clustering.md · Drain3 algorithm: project/architecture/clustering.md

Clustering

Example: id migrates to data-id inside one list

Before DOM change

  • ul > li#1
  • ul > li#2
  • Both elements in the same container and sibling set.
  • Matching relies on structure + attributes.

After DOM change

  • ul > li#1 stays unchanged.
  • ul > li[data-id='2'] replaces old li#2.
  • The cluster should stay the same: it is still one list.
  • Expected result: best candidate — li[data-id='2'].

Contract in fixtures/contract/clustering-id-migration is verified in Rust/TS/Java: list cluster preserved, heal picks li[data-id='2'] when li#2 is lost.

Element Map

Element map: flat catalog, not a DOM tree

What discover returns

  • elements — flat list of ElementNode.
  • clusters — groups linked via cluster_id.
  • No parent_id / children — traverse via filter/stream only.
  • el-N — catalog key for one run, not a permanent element ID.

How to recognize “the same” element without links

  • Structure lives in signature.path (tag:role chain to body).
  • position_in_parent — index among siblings with the same tag (not :nth-child() in the locator).
  • After an id change, heal compares signature with a new snapshot; cluster narrows search.
  • Bonus for id → data-id migration with the same value.

Element Map is a catalog for PO and baseline; cross-run identity uses Signature, not a tree in JSON. See docs/glossary.md.

Healing

Heal: how the library restores a locator

1. Locator not found
[data-testid='talk-open-healing'] missing after markup change.
2. Match signature to current page
Frap compares the saved signature with candidates in the new snapshot.
3. Pick best candidate by confidence
Best candidate: [data-testid='talk-card-open-healing'] (e.g. confidence 0.94).
4. Outcome
If confidence exceeds threshold — healed=true; otherwise safe fail with a report.

Heal is controlled locator recovery, not guessing.

Context Layer

Tri-plane context: UI + network + logs → RCA

Aggregation (F002)

  • ContextTimeline — unified event timeline.
  • UI: clicks, locator failures.
  • Network: HTTP, timings, errors.
  • Logs: console, page errors; correlation via trace_id.
  • Playwright: captureAll(true) in the adapter.

RCA — analyze_rca (F003)

  • Root Cause Analysis — post-mortem around the failure moment.
  • Core classifies: UI / network / timing / unknown.
  • Response: RcaReport (cause, confidence, excerpt, recommendation).
  • This is not heal and not discover.
  • Example: API timeout → do not “heal” the selector. Case C002.

Observability layer: explain why it failed, not only which locator to pick.

Discover vs CDP

Frap.discover today and CDP on the roadmap

Today (Playwright)

  • Frap.discover(page)
  • → SnapshotBuilder.build()
  • → page.evaluate(...) in-page JS
  • → client.buildElementMap(...)
  • CDP is not used on this path.

Planned (CDP source)

  • Standalone Chrome: URL or CDP session.
  • Scenario recording via Chrome DevTools Protocol (C009).
  • Separate transport — same Core pipeline.
  • Discover / element map / clustering without contract changes.

CDP is a future way to deliver DOM to Core; algorithms stay in the Rust core.

Conference demo: smart heal and safe refusal

✅ PASS: schedule-heal

One clear candidate — heal accepted

// ScheduleHealingTest.java FrapLocator openLink = Frap.withFrap( page.locator("[data-testid='talk-open-healing']"), page, options.minConfidence(0.7).healPolicy("expect_heal") ); openLink.click(); // Verify heal HealResult result = Frap.getLastHealResult(openLink); assertThat(result.healed()).isTrue(); assertThat(result.confidence()).isGreaterThanOrEqualTo(0.7);

❌ FAIL: CFP ambiguous

Two similar candidates — refused for safety

// CfpAmbiguousHealTest.java FrapLocator submit = Frap.withFrap( page.locator("[data-testid='cfp-submit-missing']"), page, options.minConfidence(0.85) ); // Throws — we do not click the wrong button try { submit.click(); // PlaywrightException } catch (Exception e) { HealResult r = Frap.getLastHealResult(submit); assertThat(r.healed()).isFalse(); assertThat(r.topCandidates()).hasSizeGreaterThanOrEqualTo(2); }

Two scenarios: clear heal accepted, ambiguous one rejected. Report: `frap-debug.html` with DOM clusters and top candidates.

Large projects

What matters at scale (thousands of tests)

Technical requirements

  • Snapshot caching and incremental refresh.
  • Policy modes: strict fail / controlled heal.
  • Deterministic behavior without cloud dependencies.
  • Integration with CI artifacts and reports.

Value metrics

  • Heal rate and share of ambiguous failures.
  • UI change dynamics by page and team.
  • Average selector recovery time.
  • Share of manual DOM fixes eliminated.
Architecture choice

Rust core + stack SDKs: why not duplicate algorithms

Benefits of one core

  • Single source of truth for signature/heal/clustering.
  • Same behavior in Java, TypeScript, and other SDKs.
  • Performance and determinism without drift.

Reducing vendor lock-in

  • Explicit JSON-RPC contract and DTO schema.
  • Contract tests for every SDK.
  • Thin adapters: API ergonomics without copied logic.
