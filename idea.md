## Проблематика

При автоматизации тестирования веб- и мобильных приложений команды тратят до 40% времени на поддержку селекторов и Page Object при изменениях UI. Каждый рефакторинг фронтенда ломает тесты. Существующие решения либо проприетарны, либо требуют ML/LLM, либо завязаны на один язык. Нет открытого алгоритмического решения, которое:

- Автоматически генерирует Page Object из структуры приложения
- Обеспечивает self-healing селекторов без ML
- Работает одинаково на разных языках (Java, TypeScript, Python)
- Объединяет UI, логи и сетевой трафик в одном сценарии
- Интегрируется с LLM-агентами через MCP/A2A

## Обсуждаемые варианты

**1. Архитектура ядра:**
- Чистая Java — быстро, но только одна платформа
- Rust/Go + FFI (JNI/N-API/CFFI) — единое ядро, но сложность моста
- Rust + WASM — единое ядро, запуск везде, включая браузер

**2. Расширение на логи и сеть:**
- Только UI — генератор Page Object
- UI + логи + сеть — унифицированная платформа тестирования с валидацией на трёх уровнях

**3. Мобильные платформы:**
- Только веб — ограниченный охват
- Веб + Android + iOS — единое ядро кластеризации для трёх платформ

**4. Интеграция с AI:**
- Только алгоритмический подход
- MCP/A2A-интерфейс для вызова из LLM — генерация тестов из требований, аудит, root cause analysis

**5. Модель распространения:**
- Закрытое ядро + открытые SDK — барьер для контрибьюторов
- Полностью открытый исходный код — рост сообщества, enterprise-монетизация

## К чему пришли

**Техническое решение:**
- Ядро на Rust — единая кодовая база, компиляция в WASM и нативные сборки
- Алгоритм кластеризации на основе сигнатур путей (Drain3-подход к DOM/ViewTree/логам)
- Тонкие SDK под каждый язык, генерируемые автоматически
- Единый механизм для UI (веб + Android + iOS), логов и сетевого трафика
- MCP/A2A-интерфейс для интеграции с LLM-агентами

**Продуктовое решение:**
- Два режима: запись сценария (тестировщик) и воспроизведение (CI)
- Три кейса: запись, воспроизведение, анализ
- Полностью открытый исходный код
- Монетизация через enterprise-фичи (CI-плагины, дашборды, managed-сервис)

**Организационное решение:**
- Разработка с AI/LLM (возможно, не соло — привлечение контрибьюторов через InSourceHub)
- MVP за неделю с Rust core + TS SDK + Playwright adapter
- Java SDK — позже, по запросу пользователей
- Развитие функционала по фидбеку реальных пользователей
- Продвижение через InSourceHub внутри банка как внутренний open source
- Параллельно — выход на внешний рынок (GitHub, конференции)

## Технические улучшения идеи

На основе анализа рынка и конкурентов выделены следующие улучшения:

**1. Селекторы не заменяют, а дополняют**
- Сохраняем оригинальный селектор как primary
- Добавляем fallback-цепочку на основе сигнатур
- Self-healing активируется только при failure primary

**2. Обучение на ошибках**
- Когда кластеризация находит не тот элемент, пользователь указывает правильный
- Система обновляет веса сигнатур для данного сценария
- Постепенное повышение точности под конкретное приложение

**3. Стабильность теста как метрика**
- Dashboard с "health score" каждого теста
- История self-healing событий для выявления flaky-паттернов
- Алерты при деградации стабильности

**4. Replay с вариациями**
- Воспроизведение с разными viewport (mobile/desktop)
- Throttle network (fast 3G, slow 4G)
- Проверка устойчивости теста к условиям

**5. Визуальный контекст в сигнатуре**
- Не скриншоты целиком (как в Applitools), а компактные фичи
- Относительный размер, положение в контейнере
- Стабильные стили (цвет фона, рамки)
- Улучшает точность при полной смене атрибутов DOM

**6. Playwright Adapter как первоочередная интеграция**
- Playwright быстро растёт, поддерживает все современные браузеры
- Адаптер позволяет использовать fletta без миграции с существующих тестов
- Интеграция через custom selectors API Playwright

**7. MCP-интеграция для AI-агентов**
- JSON-RPC интерфейс для вызова из LLM
- Команды: record, replay, analyze, generate
- Возможность генерации тестов из текстовых требований

**8. Тестирование самих AI-агентов**
- Capture и audit tool calls (MCP) от агентов
- Assertion-based testing для agent behavior
- A2A flow testing для мультиагентных систем
- Prompt regression testing при смене моделей
- Открывает новый рынок: QA for AI Agents

## Вывод

Создание открытой алгоритмической платформы для самоадаптирующегося тестирования — реалистичная и востребованная задача. Ключевое преимущество — объединение UI, логов и сети в одном self-healing сценарии без зависимости от ML, с возможностью интеграции с LLM-агентами. **Пересечение рынка по частям занято; wedge — OSS + explainable + tri-plane + bank on-prem.** Технология: Drain3-подход к сигнатурам DOM/логов.

**Решения по реализации:**
- **Архитектура:** Rust core + TS SDK первым, Java SDK позже
- **Скорость:** MVP за неделю; PoC-кейсы CP001–CP005 ([docs/benchmark.md](./docs/benchmark.md))
- **Продвижение:** InSourceHub внутри банка + внешний open source ([docs/audience.md](./docs/audience.md))
- **Приоритет интеграций:** Playwright adapter → JUnit export → MCP ([docs/integrations.md](./docs/integrations.md))
